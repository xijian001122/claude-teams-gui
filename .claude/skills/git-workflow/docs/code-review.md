# 代码审查

## 概述

本文档定义代码审查流程和 PR 创建规范。

## PR 创建规范

### PR 标题格式

```
<type>: <简短描述>

示例：
feat: 添加用户认证功能
fix: 修复登录超时问题
docs: 更新 API 文档
```

### PR 描述模板

```markdown
## 变更内容

- 做了什么变更
- 为什么做这个变更
- 如何解决相关 Issue

## 检查清单

- [ ] 代码自测通过
- [ ] 添加了必要的测试
- [ ] 更新了相关文档
- [ ] 遵循了项目规范

## 关联 Issue

Closes #123
Relates to #456

## 截图（如适用）

[功能截图]

## 测试说明

如何测试这个变更：
1. 步骤 1
2. 步骤 2
3. 预期结果
```

## 审查流程

### 审查者职责

1. **功能性审查**
   - 代码是否实现了预期功能
   - 边界情况是否处理
   - 是否有明显的 bug

2. **代码质量审查**
   - 代码是否清晰易读
   - 命名是否规范
   - 是否有重复代码

3. **安全性审查**
   - 是否有 SQL 注入风险
   - 是否有 XSS 风险
   - 敏感信息是否泄露

4. **性能审查**
   - 是否有性能瓶颈
   - 数据库查询是否优化
   - 是否有不必要的计算

### 审查评论规范

**建设性评论**:
```
❌ "这段代码太烂了"
✅ "这里可以考虑使用 Promise.all 来并行处理，提升性能"

❌ "为什么这样写？"
✅ "这里使用递归可能导致栈溢出，建议改用迭代方式"
```

**评论分类**:
- **阻塞性（Blocking）**: 必须修复才能合并
- **建议性（Suggestion）**: 推荐修改但不强制
- **疑问（Question）**: 需要作者解释
- **赞赏（Praise）**: 肯定好的实践

## 合并策略

### Squash Merge（推荐）

**适用场景**: 功能分支开发，提交历史较乱

**优点**:
- main 分支历史清晰
- 每个 PR 对应一个提交
- 回滚方便

### Rebase Merge

**适用场景**: 提交历史本身就很清晰

**优点**:
- 保留完整的开发历史
- 可以看到详细的修改过程

### Merge Commit

**适用场景**: 需要保留分支合并记录

**优点**:
- 明确显示哪些提交来自哪个分支

## 合并前检查清单

- [ ] CI 检查通过
- [ ] 代码审查通过
- [ ] 无冲突需要解决
- [ ] 分支已更新到最新 main

## 审查工具

### 使用 GitHub PR 功能

1. **Review Changes**: 逐行审查代码
2. **Start Review**: 开始审查，所有评论一次性提交
3. **Add single comment**: 添加单条评论
4. **Approve**: 批准合并
5. **Request changes**: 要求修改

### 本地审查

```bash
# 获取 PR 分支
git fetch origin pull/123/head:pr-123
git checkout pr-123

# 审查代码
# ...

# 测试功能
npm run test
```

## 最佳实践

### 对于作者

1. PR 要小：一个 PR 只做一件事
2. 描述要清晰：说明做了什么、为什么做
3. 及时响应：快速回复审查意见
4. 自己先审查：提交前自己先 review 一遍

### 对于审查者

1. 及时审查：收到审查请求尽快处理
2. 区分优先级：明确哪些是阻塞性问题
3. 解释原因：说明为什么需要修改
4. 保持尊重：对事不对人
