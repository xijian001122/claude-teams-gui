## Context

### 当前状态
Claude Agent GUI 已实现了 `JsonMessageCard` 组件，用于美化展示 JSON 类型的系统消息（如 `permission_request`、`task_assignment` 等）。当前实现是只读的，用户只能查看权限请求的内容，无法直接在 UI 中进行响应。

权限请求通过 inbox JSON 文件在 team-lead 和 agents 之间传递：
- `permission_request` → 发送到 team-lead 的 inbox
- `permission_response` → 发送到 agent 的 inbox

### 目标状态
用户在看到权限请求消息气泡时，可以直接点击"批准"或"拒绝"按钮完成授权，无需离开聊天界面。

## Goals / Non-Goals

**Goals:**
- 在 `permission_request` 消息卡片中渲染批准/拒绝按钮
- 点击按钮后发送 `permission_response` 到后端
- 更新 UI 显示响应状态，防止重复操作
- 后端将响应转发给请求 Agent
- 保持向后兼容，现有消息无需修改

**Non-Goals:**
- 不支持撤销已做出的授权决定
- 不修改 `permission_response` 的消息格式（保持与现有协议一致）
- 不实现复杂的权限级别（只支持完全批准或拒绝）

## Decisions

### 1. 状态存储方案：嵌入消息内容
**决策**：将授权状态（`status` 和 `response`）嵌入到 `permission_request` 消息的 JSON 内容中。

**理由**：
- 简单直观，刷新页面后状态不丢失
- 不需要额外的状态管理或数据库表
- 与现有消息存储机制兼容

**替代方案考虑**：
- 本地 useState：刷新后状态丢失 ❌
- 单独的 permissionResponses Map：需要跨组件传递，复杂 ❌

### 2. 响应后立即更新本地状态
**决策**：前端发送响应后立即更新本地消息状态，不等后端确认。

**理由**：
- 提供即时反馈，用户体验更好
- 后端写入失败的情况极少，可以依赖错误处理
- 避免用户等待时的困惑

### 3. 使用 REST API 而非 WebSocket
**决策**：使用 POST `/teams/:name/permission-response` 发送权限响应。

**理由**：
- 权限响应是一次性操作，不需要持久连接
- 与现有消息发送 API 模式一致
- 更容易处理错误和重试

**替代方案考虑**：
- WebSocket：实时性更好，但需要额外的心跳和重连逻辑 ❌

### 4. 按钮防重复点击
**决策**：点击后立即禁用按钮，并显示"处理中..."状态。

**理由**：
- 防止用户误操作重复提交
- 提供视觉反馈表示操作正在进行
- 处理完成后根据结果显示最终状态

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 网络失败导致响应未送达 | 高 | 显示错误提示，允许重试；本地状态回滚 |
| 用户误点按钮 | 中 | 添加确认对话框（可选）；明确的颜色区分 |
| 并发响应冲突 | 低 | request_id 唯一标识，后端幂等处理 |
| permission_response 不显示 | 低 | 符合预期，它是内部协议消息 |

## Migration Plan

### 部署步骤
1. 部署后端 API 变更（新增 `/teams/:name/permission-response` 端点）
2. 部署前端样式变更（新增按钮和状态徽章 CSS）
3. 部署前端组件变更（JsonMessageCard、MessageBubble、ChatArea）

### 回滚策略
- 后端变更：删除新增的路由
- 前端变更：回滚到上一版本
- 已响应的消息：保留在消息历史中，降级为纯文本显示

### 兼容性
- 旧版本消息（无 status 字段）：显示为 pending 状态，可正常响应
- 新版本消息在旧代码中：显示为普通 JSON，无交互功能

## Open Questions

1. **超时处理**：如果后端长时间未响应，是否需要超时提示？
   - 建议：30秒超时，显示"网络超时，请重试"

2. **批量授权**：是否需要支持批量批准多个权限请求？
   - 建议：暂不支持，逐个处理更清晰

3. **权限详情展开**：是否需要显示更多权限详情（如具体文件内容 diff）？
   - 建议：V1 保持简洁，点击查看原始 JSON 已有完整信息
